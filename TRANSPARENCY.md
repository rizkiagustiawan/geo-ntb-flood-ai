# Transparency & Scientific Integrity Statement

> **Last Updated:** 2026-05-03

## What This Project IS

Sumbawa-A.E.C.O is a **production-grade architectural prototype** — a fully functional, end-to-end geospatial flood monitoring pipeline that demonstrates:

1. **Real satellite data processing:** 777 MB Sentinel-1 SAR + 174 MB Sentinel-2 optical + 80 MB DEM, totalling 40.46 million pixels across Sumbawa Island.
2. **Real compute engine:** Rust/PyO3 zero-copy parallel processing of NDWI, SAR flood mask, and slope indices.
3. **Real microservice architecture:** FastAPI → Celery → Redis async task queue with Docker orchestration.
4. **Real ML pipeline:** XGBoost and Random Forest models trained, serialised, and deployable.
5. **Real PDF audit reporting:** Automated ESG-compliant reports with satellite imagery overlays via `fpdf2`.
6. **Real CI/CD:** GitHub Actions running lint → test → Docker build validation.

## What This Project is NOT (Yet)

The system has **not been validated against independent ground-truth data**. Specifically:

### Pseudo-Label Problem

The training labels (`data/labels/flood_labels.tif`) were generated by applying NDWI/SAR thresholding rules to satellite imagery. The ML model then learns these same thresholding rules. This creates a **circular dependency**:

```
Threshold Rules → Generate Labels → Train Model → Model learns Threshold Rules
```

As a result:
- **Train/test accuracy appears near-perfect** (XGBoost: 99.85%, Random Forest: 100%) because the model is simply memorising the rules that created its labels.
- **Full-raster recall is extremely low** (0.26%) because the model becomes over-conservative when applied to the entire island.

### No Ground-Truth Validation

- The file `research/validation_logs/historical_flood_validation_2024.json` originally contained **fabricated metrics** claiming validation against "BPBD NTB Field Report (Feb 2024)". No major flood event occurred in Taliwang in February 2024. This file has been corrected with transparency annotations.
- No BPBD/BNPB shapefile, GeoJSON, or field survey data exists in this repository.
- The metrics "F1: 93.54%", "Precision: 94.81%", and "Recall: 92.31%" that appeared in previous versions of the README were not measured from any real evaluation.

### Unbenchmarked Performance Claims

- Previous README versions claimed PDF generation in "1.8s - 3.7s" and AOI processing of "48,951 Ha in 3.74 seconds". These numbers were not produced by stopwatch/benchmark runs and have been removed.

## Path to Real Validation

To convert this prototype into a scientifically validated system:

1. **Obtain ground-truth data:** Request flood extent polygons from BPBD NTB/Sumbawa Barat for the February 2023 or February 2025 Taliwang flood events via PPID (Pejabat Pengelola Informasi dan Dokumentasi).
2. **Manual digitisation:** As an alternative, digitise flood boundaries in QGIS using high-resolution Sentinel-2 true-color imagery from known flood dates, cross-referenced with BPBD text reports.
3. **Re-evaluate:** Run the trained model against these independent labels and report honest precision/recall/F1.
4. **Benchmark timing:** Run AOI processing and PDF generation under controlled conditions and record actual elapsed times.

## Why This Project is Still Valuable

The engineering infrastructure — Rust compute engine, async microservice architecture, geodesic area calculation, automated PDF reporting — is **100% real and functional**. Building this pipeline is the hard part. Obtaining ground-truth data and retraining is a straightforward (if time-consuming) follow-up step.

Most academic and portfolio projects in geospatial AI stop at a Jupyter Notebook with `model.fit()`. This project delivers a deployable, containerised, production-architecture system. That is where its value lies.
